# Cursorで教えたルール、Geminiは知っていますか？AI駆動開発の「情報共有」を解決する新ツール

**【結論（Point）】**
「エディタではCursor、ターミナルではGemini CLI、ブラウザではClaudeを使うけれど、**あっちのAIに教えたルールを、こっちのAIは全然知らない…**」

AI駆動開発によって開発速度が圧倒的に向上した結果、皮肉にもチーム内（あるいは個人内の複数ツール間）での情報共有が追いつかなくなるという問題が起きています。

この問題を解決するために、すべてのAIが同じ設計思想とコンテキストを共有できる、中央集権型の共通メモ帳**「SharedMemoryServer（シェアード・メモリ・サーバー）」**をオープンソースで作りました。

---

## 1. 開発が速すぎて、AIの知識が「多重人格」化する（Reason）

今、多くのエンジニアが用途に合わせて複数のAIを使い分けています。
しかし、ここで「AI駆動開発特有の課題」に直面します。**「AI同士は記憶を共有していない」**ということです。

Cursorに「このモジュールは非推奨だよ」と教えても、Gemini CLIはそのルールを知らずに古いコードを生成し続けます。
AI駆動開発でアウトプットの量が増えるほど、AIごとの知識の分断（サイロ化）が進み、プロジェクト内でAIが**「多重人格」**のように振る舞い始めてしまうのです。


## 2. エンジニアが「AIの伝書鳩」になっていないか？（Example）

AIの多重人格化が起きると、現場ではこんな無駄な作業が発生します。

- **ルールを何度もコピペする：**
  Cursorで頑張って決めた設計のルールを、今度はClaudeのチャット欄にコピペして「これに従ってレビューして」とお願いする。
- **昨日決めた前提を忘れる：**
  「このライブラリはバグがあるから使わない」と昨日AIと決めたのに、今日別のAIツールを開いたら、またそのライブラリを自信満々に提案してくる。
- **プロジェクトが大きくなるほど破綻する：**
  教えなければいけない「チームの暗黙のルール」が増えすぎて、プロンプトに書ききれなくなる。

結局、人間であるあなたが「あっちのAIがこう言ってたよ」「こっちのAIにはこれを教えなきゃ」と、**AI同士の伝書鳩（コンテキストのルーター）**になってしまっているのです。これでは本末転倒です。

## 3. 解決策：すべてのAIが読み書きできる「黒板」を置く（Point）

この「AIごとの記憶の壁」を壊すために作ったのが、**SharedMemoryServer**です。

これは、あなたのパソコン（ローカル環境）で動く、AIのための「共有の黒板」です。最近話題の「MCP（Model Context Protocol）」という仕組みを使って、CursorやGemini CLIなど、さまざまなAIツールをこの一つの黒板に接続できます。

### どんなメリットがあるの？

1. **一回教えれば、全員が覚える**
   Cursorで考えた設計や、Gemini CLIで解決したエラーの記録が、自動的にこの黒板に書き込まれます。次にClaudeを開いたとき、Claudeはその黒板を読んでから返事をしてくれます。
2. **クラウドに依存しない（ローカルで安全・無料）**
   高額なクラウドサービス（Mem0など）を使わなくても、あなたのパソコン内の軽量なデータベース（SQLite）でサクサク動きます。社外秘のコードやルールをクラウドに送信する必要もありません。
3. **人間も読めるファイルで残る**
   AIが学んだことは、普通のテキストファイル（YAMLやMarkdown）としても保存されます。チームのメンバーにそのファイルを共有すれば、「全員のAIが最初から空気を読める状態」を作れます。

## 4. 速度に負けない「AIナレッジ基盤」を
これからのエンジニアの仕事は、毎回プロンプトに細かなルールを打ち込むことではありません。開発速度が上がる今だからこそ、ツールの壁を超えて「チームの歴史」や「設計思想」を一貫して維持できる場所が必要です。

AI間の「多重人格」化を防ぎ、ツールが入れ替わっても不変のコンテキストを維持するために、ぜひSharedMemoryServerをあなたの開発基盤（OS）に組み込んでみてください。
