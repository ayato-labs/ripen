# Ripen マネタイズ・機能セグメンテーション戦略

## 1. 基本方針 (Philosophy)
Ripenは「個人の利便性や少人数での開発体験（Single-player utility）」を**永続的に無料（OSS）**で提供し、「組織の管理・統制・スケール（Multi-player / Enterprise scale）」に対するソリューションを**有料（Commercial / SaaS）**で提供するモデルを採用します。

ユーザーの成功（チームの拡大やリモート開発への移行）が、そのまま収益化のトリガーとなる「Ngrokモデル」を志向します。

## 2. 機能の境界線 (Feature Segmentation)

### 🆓 Ripen Core (無料・OSS版)
ターゲット：個人開発者、LAN内で完結する小規模チーム（2〜5人）
提供価値：摩擦のないAI間ナレッジ共有

- **基本インフラ**: ローカルホストでの SQLite + FAISS 稼働
- **基本共有**: 同一ネットワーク内での SSE ハブ機能（LAN内での共有）
- **AI機能**: バックグラウンドでの知識蒸留（Distillation）、Maturityスコアリング
- **認証**: 簡易的なローカルファイル (`auth.json`) ベースの認証

### 💰 Ripen Enterprise / Cloud (有料版)
ターゲット：フルリモート開発チーム、中規模以上の企業、厳格なセキュリティ要件を持つ組織
提供価値：インターネット越しの安全な同期、高度な統制、インフラ管理からの解放

#### ① インターネット公開機能（Tunneling / Cloud）
- 無料版ではLAN内に閉じていたハブを、インターネット経由で安全に共有可能にする機能。
- 例: `ripen --cloud` のような1コマンドで、セキュアなトンネルを開通させ、リモートワークの同僚と即座に共有環境を構築。

#### ② エンタープライズ認証（Advanced Auth & Security）
- インターネット公開に伴う必須機能としての高度な認証。
- SAML / SSO (Google Workspace, Okta 等) 連携。
- 細かな Role-Based Access Control (RBAC)。退職者アカウントの即時無効化機能。

#### ③ 本格的データベース・グローバル同期 (Advanced DB / Scale)
- 単一の SQLite WAL の限界を超えるためのインフラ拡張。
- クラウドでの PostgreSQL 等のフルマネージドDBサポート、または Turso (LibSQL) を用いた複数拠点間（エッジ）での高速なナレッジ自動同期。
- **Value Prop**: ユーザーが自分でDBサーバーを運用・監視する手間を「ゼロ」にするマネージド体験の提供。

### 💰 料金プラン (Pricing Plan)
- **Free Plan**: 0円 (AGPL-3.0)。個人、OSSプロジェクト、教育利用。
- **Professional Plan**: 月額 1,000 円 / 年額 10,000 円。小規模チーム、商用利用。
  - 特典: 180日間の長期無料トライアル、優先サポート、商用利用権。
- **Enterprise Plan**: 個別見積もり。大規模組織、SSO必須、SLA要求。

## 3. 配信・運用スタック (Distribution & Operations)

個人開発者としての持続可能性を重視し、グローバルな税制対応やライセンス管理を自動化するスタックを採用します。

### 🔑 ライセンス管理: License-as-a-Service
- **フェーズ1 (Launch)**: [Gumroad](https://gumroad.com/)
  - **理由**: 審査なしで即日販売開始が可能。MoR（再販）モデルにより特商法の住所非公開が可能。標準の「License Key」機能を活用し、開発工数を最小化する。
- **フェーズ2 (Scaling)**: [Paddle](https://paddle.com/) または [Lemon Squeezy](https://keygen.sh/)
  - **理由**: 売上規模が拡大し、手数料（10% → 5%）のメリットが管理コストを上回った段階で移行。

### 📦 配信・リポジトリモデル (Split Repository Strategy)
知的財産の保護とOSSコミュニティへの還元を両立するため、リポジトリを物理的に分離します。

1. **Public Repository (`ripen-core`)**
   - **内容**: コアエンジン、インターフェース定義、ローカルDB実装、CLI基盤。
   - **ライセンス**: AGPL-3.0 (OSS)。誰でも利用・改善・フォーク可能。
2. **Private Repository (`ripen-enterprise`)**
   - **内容**: 有料版限定プロバイダー（Cloud/Postgres）、SSO連携、ライセンス検証ロジック。
   - **主要コンポーネント**:
     - `licensing.py`: Keygen.sh と連携したライセンス検証エンジン。
     - `plugin.py`: Core版への機能インジェクション。
   - **ライセンス**: 商用（非公開）。ソースコードの秘匿によるクラック防止。

## 4. 今後のアーキテクチャへの影響（技術的布石）
このマネタイズ戦略を将来スムーズに実行するため、オープンソース版において以下の準備を並行して進めます。
1. **DBの抽象化 (Repository Pattern)**: PostgreSQL 等のアダプターを差し込める設計。
2. **認証ミドルウェアのプラガブル化**: 将来 SSO をプラグインとして組み込めるように疎結合化。

## 5. 商用版の配信とライセンス統制 (Delivery & Enforcement)

### 📦 配信メカニズム
- **限定配布**: `ripen-enterprise` は一般公開の PyPI には登録せず、GitHub Private Repository またはプライベート PyPI で管理。
- **アクセス権の付与**: 購入者の GitHub ID に対し、リポジトリへの読み取り権限を動的に付与。

### 🛡️ ライセンスの強制力（Enforcement）
- **起動時バリデーション**: `ripen-core` の起動時に `LicenseManager` が署名を検証。 ✅ 実装済
- **暗号学的信頼**: Ed25519 公開鍵による署名検証。サーバーからのレスポンス改ざんを 100% 検知可能。
- **機能の凍結（Feature Freezing）**: ライセンス失効または不正署名時、商用機能をアンロードし OSS モードへ退避。

## 6. 法務・コンプライアンス戦略 (Legal & Compliance)

有償販売に伴う「特定商取引法（特商法）」および個人情報保護への対応方針。

### ⚖️ 特定商取引法への対応 (MoR モデルの活用)
日本国内の個人開発者が直面する「住所・氏名の公開リスク」を回避するため、**Merchant of Record (MoR)** モデルを全面的に採用する。

1. **法的販売主体の移転**:
   - 決済プラットフォーム（Lemon Squeezy 等）が「再販業者（Reseller）」として法的な販売者となる。
   - 購入者に対する領収書発行、返金対応、税務処理はプラットフォーム側が負うため、開発者の個人情報を直接決済画面に表示する必要をなくす。
2. **「特定商取引法に基づく表記」の運用**:
   - サービスサイト上に特商法に基づく表記ページを用意する。
   - 詳細は [LEGAL_TEMPLATE.md](../LEGAL_TEMPLATE.md) を参照。
   - 住所・電話番号については、**バーチャルオフィス（仮想住所）**および**IP電話/転送電話**を活用し、自宅のプライバシーを保護しつつ法的要件を満たす。
   - 「請求があった場合には遅滞なく電磁的記録等を提供します」という文言を添え、問い合わせフォーム経由での開示フローを構築する（消費者庁のガイドラインに準拠）。

### 🛡️ プライバシーと信頼性の両立
- **問い合わせ窓口の透明化**: 法人レベルのサポート用メールアドレス（例: `support@ayato-studio.ai`）を公開し、購入者がいつでも開発者と連絡が取れる状態を維持する。
- **利用規約 (EULA) の明文化**: 日本語および英語で、責任の範囲（免責事項）とライセンスの取り扱いを明記し、法的な境界線を明確にする。
