# AIエージェントに「長期記憶」と「思考ログ」を。共有メモリMCPサーバーを自作して開発体験を極限まで高める

## 序論:AIの「金魚の脳」問題、そしてもう一つの深刻な問題

CursorやClaude Desktopなど、AIエージェントを活用した開発が日常的になりました。しかし、私たちは毎日同じ壁にぶつかっています。
**「AIが、チャットを閉じた瞬間にすべてを忘れてしまう」** という問題です。

- プロジェクト特有のコーディング規約
- なぜそのアーキテクチャを採用したかという背景
- 昨日苦労して解決したバグの知見

これらを毎回新しいセッションでAIに説明し直すのは、開発者体験(DevExp)として最悪です。

しかし、もう一つ**さらに深刻な問題**があります。
それは、**一人の開発者が複数のAIエージェントを同時に使ったとき、エージェント間で認識のズレが起き、コードやアーキテクチャが壊れる** という現象です。

たとえば、こんな状況を想像してみてください。

> 1. **Antigravity** に「認証モジュールの設計方針を決めてもらう」
> 2. **Cursor** に「その方針に沿ってコードを実装してもらう」
> 3. **Claude Code** に「実装したコードのテストを書いてもらう」
> 4. **Gemini CLI** に「リファクタリングを依頼する」

これら4つのAIは、それぞれ独立したコンテキストを持っています。Antigravityが決めた「JWTではなくセッションベース認証を使う」という設計判断を、Cursorは知りません。結果として、CursorがJWTで実装し、Claude Codeはセッション前提でテストを書き、Geminiが全く別の方針でリファクタリングする――**「AIの多重人格障害」** とでも呼ぶべき混沌が生まれます。

この「金魚の脳」と「多重人格」の両方を解決するために開発したのが、**Ripen (MCP)** です。

---

## 1. Ripen(共有メモリMCP)とは?

[Model Context Protocol (MCP)](https://modelcontextprotocol.io/) を活用し、ローカル環境で動作する **「AIのための外部記憶装置」** です。

### 核となるコンセプト:「ブラックボード・アーキテクチャ」

マルチエージェントシステム(MAS)の研究分野には、**ブラックボード・モデル(黒板モデル)** と呼ばれる古典的な協調パターンがあります。複数の独立したエージェントが、全員で共通の「黒板」を読み書きすることで、直接通信せずとも一貫した行動をとれるようにする仕組みです。

`Ripen` は、この「黒板」をMCPプロトコル上に実装したものです。Antigravity、Cursor、Claude Code、Gemini CLI――どのAIエージェントからアクセスしても、**同じ記憶・同じ設計判断・同じルール** を参照できます。

### 2層の記憶構造

単なるテキストのコピペツールではありません。このシステムは以下の2層の記憶を管理しています。

1. **Memory Bank (静的記憶 / 形式知):**
   Markdownファイルとして保存され、AIが現在のモジュールの仕様やコーディングルールを読み込むためのドキュメント。人間が直接編集することもでき、Gitで差分管理が可能です。

2. **Knowledge Graph (動的・関連記憶 / 暗黙知の構造化):**
   SQLiteベースで構築され、AI自身が「Entities(エンティティ)」と「Relations(関係性)」を抽出して記録するグラフ型データベース。エージェントが開発を進めるにつれて、自動的に知識ネットワークが成長していきます。

AIはコードを書く際、事前にこのMCPサーバーを通じて `read_memory` ツールを自動で呼び出し、「今触っているコードの背景知識」を瞬時にロードしてから作業を開始します。

## 2. なぜ「複数AIの同時利用」でアルゴリズムが壊れるのか?

この問題の根本原因は、**各AIエージェントが持つコンテキストが完全にサイロ化されている** ことにあります。

```
[従来の構成] コンテキストのサイロ化

  Antigravity        Cursor           Claude Code       Gemini CLI
  ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Context A │     │ Context B │     │ Context C │     │ Context D │
  │ (独自)    │     │ (独自)    │     │ (独自)    │     │ (独自)    │
  └──────────┘     └──────────┘     └──────────┘     └──────────┘
       ↓                ↓                ↓                ↓
  設計方針を決定   → 別の方針で実装  → 不整合なテスト  → 見当違いの修正

  結果:カオス。人間が全員の「通訳パシリ」をさせられる。
```

`Ripen` を導入すると、この構造が根本から変わります。

```
[Ripen導入後] Single Source of Truth

  Antigravity        Cursor           Claude Code       Gemini CLI
  ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Agent     │     │ Agent     │     │ Agent     │     │ Agent     │
  └────┬─────┘     └────┬─────┘     └────┬─────┘     └────┬─────┘
       │                │                │                │
       └────────────────┴────────┬───────┴────────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │   Ripen    │
                    │ ========================│
                    │  Knowledge Graph (SQLite)│
                    │  Memory Bank (Markdown)  │
                    │  Thought History (SQLite) │
                    │  Audit Logs              │
                    └─────────────────────────┘

  結果:全員が同じ「楽譜」を見て演奏する。人間は「指揮者」になれる。
```

### 具体的にどう防ぐのか?

1. **設計判断の永続化:**
   Antigravityが「セッションベース認証を採用する」と決めた際、その判断は Knowledge Graph に `Entity: AuthModule, Observation: "セッションベース認証を採用。理由:..."` として保存されます。

2. **作業前の自動ロード:**
   Cursorが認証モジュールの実装を始めると、MCPを通じて `read_memory("認証")` が呼ばれ、先ほどの設計判断が**コンテキストとして自動注入**されます。

3. **矛盾の自動検出:**
   もしClaude Codeが「JWT認証に切り替える」という矛盾する知識を書き込もうとした場合、`check_conflict` 機能がGeminiを使って既存知識との矛盾を検出し、`conflicts` テーブルに記録します。

4. **監査ログによる追跡:**
   すべてのエージェントの読み書き操作は `audit_logs` テーブルに記録されるため、「誰が(どのAIが)、いつ、何を変更したか」を完全に追跡できます。

## 3. ただのRAGじゃない。「思考(Thought)」の蒸留と自己組織化

このシステムの最大の特徴は、**「AIの思考プロセス自体を記録・蒸留する機能」**を持っていることです。

### Sequential Thinking(逐次思考)の保存
複雑な問題を解く際、AIにすぐコードを書かせるのではなく、一歩ずつ論理を組み立てさせるアプローチが有効です。このMCPは、AIの各思考ステップ(Thought)を独立したデータベースに記録します。

### 思考から知識への「自動蒸留(Distillation)」
セッションが終了すると、AIは書き溜めた思考ログを自分で振り返り、**「次回以降も使える汎用的な知識」だけをKnowledge Graphに抽出・保存**します(暗黙知から形式知へのカプセル化)。

つまり、あなたがAntigravityと壁打ちして解決したバグの知見は、Cursorが翌日コードを書くときに**自動的に参照できる「共有資産」**に変換されます。もう二度と「昨日のAIには説明したのに...」と嘆く必要はありません。

## 4. 【技術的見どころ】aiosqliteによる「ハーネス」の強靱化

このシステムはローカルのSQLiteをデータベースとして使用しています。しかし、複数のAIエージェントを同時に動かすと、超高速かつ並列にこのMCPサーバーにアクセスしてきます。

従来の `sqlite3` ライブラリ(同期処理)では、複数のAIが同時に思考を書き込もうとした際に `database is locked` エラーが頻発し、システムが破綻していました。

### async(非同期)への全面移行
この課題を解決するため、バックエンドを **`aiosqlite`** を用いた完全な非同期アーキテクチャに移行しました。

```python
# database.py の一部(非同期化されたコネクション管理)
import aiosqlite
import asyncio
import random

def retry_on_db_lock(max_retries=15, initial_delay=0.1):
    def decorator(func):
        async def wrapper(*args, **kwargs):
            retries = 0
            while retries < max_retries:
                try:
                    return await func(*args, **kwargs)
                except aiosqlite.OperationalError as e:
                    if "database is locked" in str(e).lower():
                        retries += 1
                        if retries == max_retries:
                            raise e
                        # Exponential backoff with jitter
                        delay = min(
                            initial_delay * (2 ** (retries - 1)), 1.0
                        ) + random.uniform(0, 0.1)
                        await asyncio.sleep(delay)
                    else:
                        raise
            return await func(*args, **kwargs)
        return wrapper
    return decorator


async def async_get_connection():
    conn = await aiosqlite.connect(get_db_path(), timeout=30.0)
    conn.row_factory = aiosqlite.Row
    await conn.execute("PRAGMA foreign_keys = ON")
    await conn.execute("PRAGMA journal_mode = WAL")
    return conn
```

この「Exponential Backoff(指数的後退)+ Jitter」を組み合わせた非同期ロック再試行メカニズムと、WALモード(Write-Ahead Logging)の組み合わせにより、**複数のAIエージェントが同時にアクセスしても耐えうる堅牢な「ハーネス(馬具)」**が完成しました。

AI(強力な馬)を制御し、知識を安全に蓄積するための**「ハーネス・エンジニアリング」**――モデルそのものではなく、モデルを支える基盤のエンジニアリングこそが、マルチエージェント時代の要です。

## 5. なぜSaaSではなく「ローカルMCP」で作ったのか?

クラウドのベクトルデータベース(Pinecone等)を使えば簡単なのですが、あえて「ローカルのSQLite + Markdown」で構築したのには理由があります。

- **プライバシーとセキュリティ:** 会社の機密コードのコンテキストを外部SaaSに送信したくない。データは自分の手元にある。
- **データ主権:** 知識はローカル環境にあり続け、Gitで差分管理できる。ベンダーが倒れても知識は失われない。
- **ベンダーロックインの回避:** MCPというオープン標準に乗ることで、Claude DesktopでもCursorでもGemini CLIでも、**ツールを問わずに同じ記憶を使い回せる**。これこそが「複数AIの同時利用」を可能にする鍵。
- **ゼロコスト:** 追加のSaaS契約は不要。SQLiteとMarkdownという、人類が最も信頼してきた技術スタックのみで構成されている。

## 6. 今後の課題:AI自身の進化か、システムの功績か?

開発を進める中で、一つ避けては通れない「贅沢な悩み」に直面しています。それは、**「AIが賢くなったのは、このMCPのおかげなのか、それとも単純にAIモデル自体の性能が上がっただけなのか?」** という問いです。

昨今のGeminiやClaudeの進化スピードは凄まじく、モデル単体でも驚くほど高度な推論を行います。自作システム開発者としては、自分の「ハーネス」がどれだけ寄与しているのかを正確に測ることは極めて困難です。

そこで現在、以下の**4つの観点**からシステムの有効性を検証・改善しようとしています。

1. **アブレーション・スタディ(切除研究):**
   同じ課題を「MCPあり」と「MCPなし」で解かせ、回答の質や正確性の差を比較する。
2. **「説明コスト」の計測:**
   人間がプロンプトで説明しなければならない前提知識が、MCP導入後にどれだけ削減されたか。
3. **監査ログ(audit_logs)の追跡:**
   AIの回答が、実際にDBから読み取ったどの「知識」に基づいているのかをトレースする。
4. **設計の一貫性(Decay Rate)の観測:**
   長期プロジェクトにおいて、初期の設計方針がどれだけ長く守られ続けているかを観測する。

「魔法のように賢くなった」と喧伝するよりも、このように**「知能の寄与分を可視化しようとする試み」**こそが、マルチエージェント時代のインフラ開発には求められていると感じています。
