# MCPツール役割定義書 (MCP Tool Roles and Use Cases)

## 1. ツール設計の哲学：能動的想起と情報の分離

`SharedMemoryServer` は、AIエージェントのコンテキストウィンドウを「量」で埋めるのではなく、情報の「質」と「論理構造」で最適化することを目指しています。そのため、全てのツールは以下の2段階検索（Search-then-Fetch）パターンに基づいて設計されています。

1.  **Discovery（発見）**: 膨大な知識の中から「何があるか」を高速かつ軽量に特定する。
2.  **Acquisition（取得）**: 特定された知識の中から「今必要なもの」だけを詳細に読み込む。

---

## 2. ツール・レイヤー定義

各ツールは、その目的に応じて以下のレイヤーに分類されます。

### 2-1. 探索レイヤー (Discovery Layer)
知識のインデックスを走査し、関連する情報の「存在」を特定します。

- **`search_memory(query)`**
    - **役割**: クエリに関連する知識のIDと要約をリストアップ。
    - **ユースケース**: 「〇〇に関する過去の決定事項はあるか？」など、情報の有無を確認する最初のアクション。
- **`list_inactive_knowledge()`**
    - **役割**: 現在アーカイブされている（非アクティブな）知識のリストを取得。
    - **ユースケース**: 過去に忘却された知識を再活性化させる必要がある場合の調査。

### 2-2. 取得レイヤー (Acquisition Layer)
特定されたIDに基づき、具体的な内容を取得します。

- **`fetch_details(ids)`**
    - **役割**: 指定した複数のIDに対応する詳細内容を一括取得。
    - **ユースケース**: `search_memory` で見つけた候補の中から、特に重要な3件の詳細を読み込む。
- **`read_memory(query)`** [Legacy/Deprecated]
    - **役割**: 検索と詳細取得を一度に行う。
    - **注意**: コンテキストを大量に消費するため、原則として `search_memory` + `fetch_details` の組み合わせを推奨。

### 2-3. 思考レイヤー (Reasoning Layer)
知識の取得と推論、および推論結果の定着を同時に行います。

- **`sequential_thinking(...)`**
    - **役割**: 過去の思考ログ（Salvage）を参照しながら多段階推論を行い、結果を自動的に知識化（Accretion）する。
    - **ユースケース**: 複雑な設計判断、長期間にわたるデバッグ、新しい要件の実装方針策定。

### 2-4. キュレーションレイヤー (Curation Layer)
知識ベースの品質を維持・向上させます。

- **`save_memory(...)`**
    - **役割**: 新しいエンティティ、観測、関係、またはファイルを保存。
    - **ユースケース**: 作業の節目での進捗記録、新しい技術スタックの学習内容の保存。
- **`synthesize_entity(entity_name)`**
    - **役割**: 特定のエンティティに関する全知識をAIが要約。
    - **ユースケース**: 「このプロジェクトのこれまでの経緯を1分で教えて」という要求への回答生成。
- **`manage_knowledge_activation(ids, status)`**
    - **役割**: 知識の有効/無効を切り替え。
    - **ユースケース**: 完了した一時的なタスクの情報を非アクティブ化して検索ノイズを減らす。

## 3. 標準的なワークフロー例 (Scenarios)

### シナリオA：過去の不具合対応策を調査する
1.  `search_memory(query="Memory Leak in asyncio")`
2.  返ってきたリストから関連性が高い2件のID（`entity:LeakFix_2024`, `bank:debug_log_05.md`）を選択。
3.  `fetch_details(ids=["entity:LeakFix_2024", "bank:debug_log_05.md"])`
4.  詳細を確認し、解決策を適用。

### シナリオB：複雑な新機能の設計を行う
1.  `sequential_thinking(thought="Implementing plugin system architecture")` を開始。
2.  自動的に提示される `related_knowledge`（過去の設計原則の要約）を確認。
3.  必要に応じて `fetch_details` で過去のプラグイン実装例の詳細を取得。
4.  思考を継続（`thought_number=2, 3...`）。
5.  最後のステップで `next_thought_needed=false` とし、設計方針をコアメモリへ自動蒸留（Accretion）。

---

## 4. 運用上のヒント

- **詳細の取得を最小限にする**: AIエージェントには、まず要約を読んでから詳細を取りに行くようにプロンプトで指示してください。
- **思考の永続化**: 単発の検索ではなく `sequential_thinking` を使うことで、なぜその結論に至ったかという「思考の軌跡」自体が将来の検索対象になります。

## 付録: 管理・保守レイヤー (Admin Tools)

システムアーキテクチャの統合に伴い、管理用機能も同一のサーバーに `admin_` プレフィックス付きで実装されています。これらは主に人間の開発者や、自己メンテナンスを行う自律エージェントによって使用されます。

- **ポータビリティとバックアップ (`admin_export_knowledge`, `admin_import_knowledge`)**
    - **役割**: データベース全体を人間可読な YAML 形式でエクスポート / インポートします。バイナリスナップショットに代わる、安全でバージョン管理可能な移行手段です。
- **整合性チェック (`admin_check_integrity`)**
    - **役割**: DB破損、インデックスのズレ、孤立したエンティティのチェックを行います。
- **インデックス再構築 (`admin_rebuild_index`)**
    - **役割**: 全ての埋め込み（Embedding）と検索インデックスの強制的な再構築を行います。インポート後などに実行します。
- **ガベージコレクション (`admin_run_knowledge_gc`)**
    - **役割**: 一定期間アクセスのない、または価値の低い古い知識をクリーンアップします。
