# 概念的要件定義書 (Conceptual Requirements Definition)

SharedMemoryServerは中央集権型でローカルホストのMCPサーバーによるナレッジ共有システム。
解決するイシューは、「AI駆動開発によって圧倒的に開発速度が上がったことで、チーム開発の際の情報共有がより難しくなったこと」（個人で複数AIエージェント間の知識の共有にも使用可能だけど。）
計画策は、中央集権型のナレッジ共有のMCPサーバーを立てること。

## 1. プロジェクトのビジョン
`SharedMemoryServer` は、AIエージェントが長期的な「自己」や「思考の文脈」を維持するための、高信頼・低レイテンシな記憶インフラストラクチャを提供します。



### 背景と目的
現代のAIワークフローにおける最大のボトルネックは、コンテキストウィンドウのサイズではなく、情報の**「減衰（Decay）」**と**「エントロピー（増大する無秩序）」**です。
単なるベクター検索（Naive RAG）では、設計上の意思決定や論理的な繋がりといった「構造化された知識」を維持できません。本プロジェクトは、エージェントが過去の自分の思考を「再構築」し、セッションを跨いで一貫した論理を展開できるようにすることを目的としています。

## 2. 実装方針（何を、なぜ行うのか）

> [!NOTE]
> 各ツールの具体的な役割、分類、および標準的なワークフローについては、[MCPツール役割定義書](MCPツール役割定義書.md) を参照してください。

### 2-1. Graph（構造）と Bank（非構造）のハイブリッド
- **方針**: 知識を「エンティティ・リレーション」によるグラフ構造と、「Memory Bank（Markdownファイル群）」による深いテキストデータの両方で管理します。
- **理由**: 
    - グラフは「AはBに依存する」といった明確な**論理構造**を保持するのに適しています。
    - バンクは、背景知識や複雑な技術仕様といった、グラフ化しきれない**深いコンテキスト**を保持するのに適しています。
    - これらを組み合わせることで、エージェントは「事実の繋がり」と「詳細な背景」を同時に取得できます。

### 2-2. SSoT (Single Source of Truth) とローカルファースト・プロバイダー
- **方針**: SQLite (FTS5) と FAISS を組み合わせ、全ての状態をローカルのファイルベースで永続化します。さらに、LLM および Embedding エンジンは「ローカルファースト」をデフォルト（FastEmbed + Ollama）とし、外部 API への依存をオプション化します。
- **理由**: 
    - **トレーサビリティとポータビリティ**: 開発者やAI自身がいつでもデータを人間が読める形（YAML等）で検査・編集でき、監査やマイグレーションが容易になります。
    - **低摩擦とゼロ・レイテンシ**: 外部のベクターDBサーバーやクラウド API を待つ必要がなく、検索レイテンシを **< 20ms** まで短縮することで、AI の推論ループを妨げません。
    - **完全なプライバシー**: デフォルト構成では、企業の機密コードや設計思想が外部サーバー（Google 等）に送信されるリスクを 100% 排除します。
    - **信頼性**: ネットワーク遅延や外部サービスの停止、レート制限のリスクを排除し、オフライン環境でも一貫した動作を保証します。

### 2-3. Compute-then-Write パターン
- **方針**: 埋め込み（Embedding）計算やコンフリクト検知などの重いAI処理を、データベースのトランザクションの外で実行します。
- **理由**: 
    - マルチエージェント環境において、SQLのロック時間を最小化するためです。
    - これにより、DBのロック時間を秒単位からミリ秒単位（<50ms）へ短縮し、システム全体の並行性を確保します。

### 2-4. 知識の「熟成」と「減衰」の管理
- **方針**: アクセス頻度（活用実績）と最終参照からの経過時間に基づき、知識の状態（アクティブ/非アクティブ）を管理します。
- **理由**: 
    - 全ての情報を等しく保持し続けると、検索のノイズが増大します。
    - 頻繁に参照される「重要な設計原則」を**長期資産**として際立たせ、一定期間アクセスのない「一時的なノイズ」を自動的にアーカイブ（GC）することで、記憶の質を高く保ちます。

### 2-5. Sequential Thinking の統合と相乗効果
- **方針**: 推論プロセス（Sequential Thinking）を単なる一時的な出力ではなく、永続化された「思考ログ」として管理し、コアメモリと相互作用させます。
- **理由**: 
    - **Salvage（サルベージ）**: 推論の各ステップにおいて、関連する過去の知識の**要約とID**を自動的に検索し、エージェントに提示します。詳細が必要な場合、エージェントは自ら `fetch_details` ツールを使用してその知識を取得します。これにより、コンテキストの肥大化を防ぎつつ、エージェントの意思に基づいた「能動的な想起」を促します。
    - **Accretion（蓄積/蒸留）**: 推論セッションから重要な知見を抽出し、自動的に Graph や Bank へ反映（蒸留）します。増分抽出時には「既存のコンテキスト」を考慮させ、冗長な知識生成を防ぎます。抽出結果はJSONバリデーションによって厳格に検証され、スキーマ整合性を担保します。
    - **Reasoning Provenance**: 「なぜその結論に至ったのか」という思考の軌跡を永続化することで、セッションを跨いだ論理の一貫性維持と、後からの論理監査を可能にします。

### 2-6. 知識の不整合（コンフリクト）の検知と人間による承認 (HITL)
- **方針**: 新しい情報が既存の知識と矛盾する場合、AI が自動的に不整合を検知し、即座に主記憶へ書き込むのではなく「承認待ち（Conflict Center）」へステージングします。
- **理由**: 
    - **SSoT (Single Source of Truth) の品質維持**: 誤った情報や一時的なハルシネーションによって、チームで共有される「黄金の知識ベース」が汚染されるのを防ぎます。
    - **人間による最終判断**: 知識の「進化（方針変更）」なのか「誤り（ノイズ）」なのかの最終的な判別を人間に委ねることで、システムの信頼性を担保します。
    - **透明性の確保**: ダッシュボードを通じて、AI が何を「矛盾」と判断したかを可視化し、知識の更新プロセスをブラックボックス化させないようにします。

### 2-7. コンテキストの密度最適化（Signal-to-Noise Ratio）
- **方針**: MCPツールを介した検索結果の返却数を、常に推論に最適な「Top 10〜20件」程度に厳選し、必要に応じて内容を要約・切り出し（Truncation）します。
- **理由**: 
    - **推論精度の維持**: 大量の情報を一度に提示すると、LLMの「情報の埋没（Lost in the Middle）」を引き起こし、推論精度が低下します。高シグナルな情報のみを届けることで、エージェントの判断能力を最大化します。
    - **システム安定性**: 通信レイヤーでのタイムアウトやメモリ不足を防ぎ、商用グレードの安定性を確保します。
    - **コストと速度**: トークン消費量を抑制し、レスポンスのレイテンシを最小化します。

### 2-8. ドキュメント戦略：日英併記（English Primary + 日本語セクション）
- **方針**: README をはじめとする主要ドキュメントは、英語をメイン構造としつつ、日本語の専用セクションを併設する構成を採用します。
- **理由**: 
    - **発見可能性（Discoverability）**: GitHub の検索エンジンや `awesome-mcp-servers` リスト、海外コミュニティからの流入を確保するため、公用語である英語をベースとします。
    - **共感と転換率（Conversion）**: 日本のAI駆動開発者に対し、冒頭の日本語メッセージで「自分の課題を解決するツールだ」という直感的な共感（AHA体験）を提供します。
    - **成功パターンの踏襲**: `Deno` や `Localsend` など、グローバルで成功しているOSSが採用している「英語メイン＋母国語サポート」のハイブリッドモデルに準拠します。

### 2-9. マルチエージェント認証と変更履歴（Audit）
- **方針**: 複数のAIエージェントやツールが同一の SharedMemoryServer を共有する際、「どのエージェントが、いつ、何を記録したか」を正確に特定し、ログとして残します。
- **実装**:
  - `data/auth.json` による API キーベースの認証ゲートウェイ。
  - クライアントごとに割り当てられた `SHARED_MEMORY_ACCOUNT` を `audit_logs` および `thought_history` に紐付け。
  - セキュリティと利便性のバランスを考慮し、環境変数によるAPIキーの注入を標準とする。
- **理由**: 
  - **責任の明確化**: AIエージェントが誤った情報を記録した場合、どのセッション・どのエージェントに起因するかを追跡可能にすることで、デバッグと知識の修正を容易にします。
  - **チーム開発の基盤**: 将来的に複数の人間開発者が異なるAIツールを併用するチーム環境において、知識の「出所（Provenance）」を保証します。

### 2-10. 定量的評価指標：LongMemEval
- **方針**: 記憶の「想起精度」と「推論レイテンシ」を定量的に評価するための独自のベンチマークスイート `LongMemEval` を運用します。評価ロジックには、業界標準である **RAGAS (RAG Assessment)** のメトリクスを採用します。
- **理由**: 
    - **客観性と信頼性の担保**: 独自のスコアリングではなく、RAGAS（Faithfulness, Context Recall 等）を用いることで、外部からも納得感のある品質評価を可能にします。
    - **デグレードの早期発見**: プロバイダー（Gemini/Ollama）やモデルを変更した際に、記憶の検索精度が低下していないかを客観的に検証します。
    - **パフォーマンスの最適化**: 12ms 級の超低レイテンシを維持しつつ、Recall@10（上位10件の正解率）を 90% 以上に保つための設計ガイドラインとして機能させます。

### 2-11. 動的なコンテキスト圧縮とトークン管理 (Dynamic Context Compression)
- **方針**: LLMのコンテキストウィンドウ制限（特に500/413エラーの要因となるペイロード過大）を防ぐため、API送信前にトークン数を事前計算し、制限に近づいた場合は自律的に「情報の蒸留（Distillation）」を行う圧縮レイヤーを設けます。
- **実装**:
  - `config.py` にて、推論用（Generation）と圧縮用（Compression）のモデルプールを独立して定義・管理。
  - プロンプト送信前に `count_tokens` を用いてサイズを評価し、安全閾値（例: 制限の90%）を超過した場合、圧縮用モデルによる再帰的な要約処理を自動でトリガー。
- **理由**: 
    - **システムの堅牢性（Resiliency）**: 情報量が増大してもサーバーがクラッシュすることなく、エラーを自己修復して処理を継続できるようにするため。
    - **意味の保存（Semantic Preservation）**: 単純な文字数カット（Truncation）ではなく、LLM自身に「事実の抽出」をさせることで、重要な文脈や論理構造を維持したまま安全なトークンサイズへとダウンサイズします。
    - **コストと速度の分離**: 将来的に「圧縮には高速・安価なモデル（Flash等）」、「推論には高精度モデル（Pro等）」といった戦略的なルーティングを可能にするため。

### 2-12. 思考プロセスの厳格なID管理と整合性保証 (Thought Integrity)
- **方針**: Sequential Thinking における各思考ステップ（`thought_number`）を、セッション内で一意（Unique）な識別子として厳格に管理し、意図しない重複やデータの不整合を物理的に遮断します。
- **実装**:
  - **データベース制約**: `thought_history` テーブルの `(session_id, thought_number)` に対して `UNIQUE` インデックスを適用し、DBレベルでの二重登録を防止。
  - **多重防御（Defense in Depth）**: アプリケーション層で挿入前に既存IDをチェックし、重複時には詳細なエラーメッセージ（修正なら `is_revision=True` を使用せよ、等）をエージェントに返却。
  - **下位互換性の維持**: 既存の重複データがある環境でも起動を妨げないよう、マイグレーション時には警告付きのフォールバック・ロジックを採用。
- **理由**: 
    - **推論の完全性（Provenance）**: 同じ番号の思考が複数存在すると、修正（Revision）や分岐（Branching）の対象が不明確になり、推論の軌跡が破壊されるため。
    - **メタデータの保護**: 重複IDによる上書きが発生すると、過去の思考に紐付いていた検索結果や統計情報といったメタデータが汚染されるリスクを排除するため。
    - **エージェントへのフィードバック**: 「状態管理の誤り」を即座にエラーとして返すことで、AIエージェント自身に正しいプロトコルへの自己修正を促し、より高度な自己組織化を実現するため。

## 3. 非目標（何を、なぜ行わないのか）

### 3-1. 自律的な推論（Reasoning）の実装
- **非目標**: サーバー自体が知識を組み合わせて新しい推論を行うことはしません。
- **理由**: 本サーバーは「海馬（記憶を司る部位）」に特化すべきであり、「前頭前野（思考を司る部位）」は呼び出し側（エージェント）の責任であるためです。責任の分離（SoC）を徹底します。

### 3-2. ペシミスティックな書き込み拒絶（ゲートキーピング）
- **非目標**: 既存の知識との矛盾を理由に、新しい情報の保存を拒絶（ブロック）することはしません。
- **理由**: 
    - **進化の阻害防止**: サーバーは「真偽の絶対的な審判」ではなく、あくまで「観測の記録者」であるべきだからです。情報の正当なアップデートや方針転換をシステムが妨げるリスクを排除します。
    - **パフォーマンスの優先**: 書き込み時の厳密なバリデーションはレイテンシを増大させます。矛盾は「エラー」として排除するのではなく、後続の蒸留プロセスやエージェントの推論において「解決すべき課題」として提示する方針をとります。

### 3-3. グラフ構造の動的な可視化
- **非目標**: リアルタイムでノードとエッジが動く複雑なネットワークグラフ表示機能は持ちません。
- **理由**: インフラとしての機能性に注力するためです。可視化が必要な場合は、ダッシュボード上のリスト形式や、外部ツールへのエクスポート機能を優先します。

### 3-4. ヒューリスティックな重み付け検索
- **非目標**: 「"Error" という文字が含まれていたらスコアを+10する」といった、ハードコードされた経験則（ヒューリスティック）は実装しません。
- **理由**: 現代の埋め込みモデルは十分に文脈を理解しています。手動の重み付けは将来のモデル改善の恩恵を阻害し、メンテナンスコスト（技術負債）を増大させるためです。

### 3-5. 複雑なマルチユーザー権限管理
- **非目標**: 粒度の細かいユーザー別アクセス制御（RBAC）などは実装しません。
- **理由**: 本サーバーは個人、あるいは特定のプロジェクトチームの「専属記憶」としての利用を想定しています。過度な複雑さはパフォーマンスとポータビリティを損なうためです。

### 3-6. 無制限なデータ返却（Unbounded Retrieval）
- **非目標**: 「過去の全履歴を一度にリストアップする」といった機能を、エージェントが利用するMCPツールとしては提供しません。
- **理由**: 
    - **役割の分離**: MCPツールは「推論とアクション」のための実行レイヤーであり、膨大なデータの監査や俯瞰的な分析は、別途構築される「管理用UI（管理レイヤー）」の責務であるためです。
    - **認知負荷の制御**: 全履歴の提示はエージェントにとってのノイズとなり、目的の達成を阻害するリスクがあるためです。

### 3-7. 完全非同期化による極限のレイテンシ最適化 (Over-Optimization for Latency)
- **非目標**: セッション終了時の最終的な知識蒸留（Final Distillation）において、エージェントへのレスポンス速度を数秒縮めるためだけの「完全な非同期化（バックグラウンドへの完全分離）」は行いません。
- **理由**: 
    - **Read-After-Writeの一貫性**: エージェントが思考を終えた直後のタスクで過去の記録を検索した際、「さっき結論づけたはずの知識」がまだDBに反映されていない状態（ハルシネーションの誘発）を防ぐため、セッションの境界では確実に状態を同期させます。
    - **ライフサイクルの安全性**: MCPサーバーが一時的なプロセスとして実行される環境において、バックグラウンドタスクが完了する前にプロセスが終了（SIGTERM等）し、知識が永久にロストするリスクを排除します。数秒のレスポンス向上よりも、データの確実な永続化（信頼性）を優先します。

### 3-8. 特定のクラウドベンダーへのロックイン
- **非目標**: Google Gemini 専用のシステムとして固定することはしません。
- **理由**: ユーザーのセキュリティポリシーやコスト、速度の要件に合わせて、ローカル LLM (Ollama) や将来的な他社 API へ柔軟に切り替えられる「プロバイダー非依存（Agnostic）」なアーキテクチャを維持するためです。

## 4. 設計哲学
**「複雑さよりも信頼性、機能数よりも整合性」**

エージェントが迷ったとき、常に「そこに行けば正しい過去の記録がある」と確信できる場所であることが、SharedMemoryServer の唯一の存在意義です。
