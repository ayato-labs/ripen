# 概念的要件定義書 (Conceptual Requirements Definition)

## 0. プロジェクト概要

**Ripen** は、ローカルホストのSSEサーバーを起動することで、チームの全AIエージェント間の知識を自動的に共有できる、中央集権型MCPナレッジ共有インフラです。

**核心価値**: Claude Code、Cursor、Antigravity、Gemini CLI——違うアカウントを使った別の人のPCで稼働するAIエージェントとの間でも、知識を共有できる。

解決するイシュー: **「AI駆動開発によって開発速度が上がったことで、チーム内・製品間・パーソン間の情報共有がより難しくなった」こと。**

解決策: **中央集権型のSSEハブ型MCPサーバーを立て、全エージェントが同一の知識ベースを読み書きする。**

### **価値フロー**

[./sequential_thinkingを取り巻く一連のフローと詳細アルゴリズム.md](./sequential_thinkingを取り巻く一連のフローと詳細アルゴリズム.md)
がRipenの価値を生み出し、ローカルホストを経由する計画がそれを増幅するっていうのがRipenの設計

## 1. プロジェクトのビジョン

`Ripen` は、AIエージェントが長期的な「自己」や「思考の文脈」を維持するための、高信頼・低レイテンシな**チーム共有記憶インフラストラクチャ**を提供します。

### 核心的な差別化ポイント

一般的なMCPメモリサーバーは `stdio` モードで動作し、**1:1接続**（1つのIDEと1つのサーバーのみ）です。知識はそのIDEプロセスの中に閉じており、他のエージェントや他のユーザーと共有できません。

**Ripenは `SSEハブ` として動作します。** HTTPサーバーとして常駐し、複数のIDE・複数のユーザーが**同時かつ自動的に**同一の知識ベースを共有します。

**最重要な点**: この知識共有は、同じ人間が使う複数のAIツール間だけでなく、**別のPC・別のアカウントで稼働する別のユーザーのAIエージェントとの間でも実現できます。**

```
[構造的な差別化の図]
  Developer A のPC (Cursor, Antigravity)
  Developer B のPC (Claude Code, Gemini CLI) ← 違うアカウント・違うマシン
  CIサーバー (GitHub Actions Agent)
        ↓ ↓ ↓ ↓
      Ripen SSE Hub (共有知識ベース)

  エージェントフレームワークはこれを実現できない：
  LangChain / CrewAI は同一フレームワーク内のみ。別会社のツール間は不可能。
  MCPプロトコル層にあるRipenのみが、この壁を超えられる。
```

### 背景と目的
現代のAIワークフローにおける最大のボトルネックは、コンテキストウィンドウのサイズではなく、情報の**「減衰（Decay）」**と**「エントロピー（増大する無秩序）」**です。さらに複数エージェント・複数ユーザー環境では、**「知識の分断（Siloing）」**が加わります。本プロジェクトは、これらすべてを解決し、エージェントがセッションを跨いで一貫した論理を展開できるようにすることを目的としています。

## 2. 実装方針（何を、なぜ行うのか）

> [!NOTE]
> 各ツールの具体的な役割、分類、および標準的なワークフローについては、[MCPツール役割定義書](MCPツール役割定義書.md) を参照してください。

### 2-1. Graph（構造）と Bank（非構造）のハイブリッド
- **方針**: 知識を「エンティティ・リレーション」によるグラフ構造と、「Memory Bank（Markdownファイル群）」による深いテキストデータの両方で管理します。
- **理由**: 
    - グラフは「AはBに依存する」といった明確な**論理構造**を保持するのに適しています。
    - バンクは、背景知識や複雑な技術仕様といった、グラフ化しきれない**深いコンテキスト**を保持するのに適しています。
    - これらを組み合わせることで、エージェントは「事実の繋がり」と「詳細な背景」を同時に取得できます。

### 2-2. SSoT (Single Source of Truth) とローカルファースト・プロバイダー
- **方針**: SQLite (FTS5) と FAISS を組み合わせ、全ての状態をローカルのファイルベースで永続化します。さらに、LLM および Embedding エンジンは「ローカルファースト」をデフォルト（FastEmbed + Ollama）とし、外部 API への依存をオプション化します。
- **理由**: 
    - **トレーサビリティとポータビリティ**: 開発者やAI自身がいつでもデータを人間が読める形（YAML等）で検査・編集でき、監査やマイグレーションが容易になります。
    - **低摩擦とゼロ・レイテンシ**: 外部のベクターDBサーバーやクラウド API を待つ必要がなく、検索レイテンシを **< 20ms** まで短縮することで、AI の推論ループを妨げません。
    - **完全なプライバシー**: デフォルト構成では、企業の機密コードや設計思想が外部サーバー（Google 等）に送信されるリスクを 100% 排除します。
    - **信頼性**: ネットワーク遅延や外部サービスの停止、レート制限のリスクを排除し、オフライン環境でも一貫した動作を保証します。

### 2-3. Compute-then-Write パターン
- **方針**: 埋め込み（Embedding）計算やコンフリクト検知などの重いAI処理を、データベースのトランザクションの外で実行します。
- **理由**: 
    - マルチエージェント環境において、SQLのロック時間を最小化するためです。
    - これにより、DBのロック時間を秒単位からミリ秒単位（<50ms）へ短縮し、システム全体の並行性を確保します。

### 2-4. 知識の「熟成度（Maturity）」に基づいた階層化管理
- **方針**: 知識をその確実性と検証状況に応じて `STABLE`（検証済み）、`OBSERVED`（観測済み）、`TRANSIENT`（一時的）の3段階に分類し、検索時の重み付けを動的に変更します。
- **階層定義**:
    - **STABLE (High Signal)**: `troubleshooting_knowledge`（トラブルシューティング解決策）や `entities`（定義済みエンティティ）。最も高い検索ブースト（1.5x〜3.0x）が適用されます。
    - **OBSERVED (Context)**: `bank_files`（ドキュメント）や `observations`（事実の断片）。標準的な重み付けで扱われます。
    - **TRANSIENT (Process Noise)**: `thought_history`（推論の過程）。推論の連続性を保つために保持されますが、検索時には大幅なペナルティ（0.3x）が課せられ、自動サルベージ（Salvage）からは除外されます。
- **理由**: 
    - 過去の「試行錯誤（ノイズ）」と「導き出された結論（シグナル）」を同一視すると、AIが過去の自分の迷いに引きずられる「エコーチェンバー現象」が発生します。
    - 重要な設計原則やバグ修正方法を**長期資産**として最優先で想起させ、一時的な思考プロセスを背景へと退けることで、記憶の質を最大化します。

### 2-5. Sequential Thinking の統合と相乗効果
- **方針**: 推論プロセス（Sequential Thinking）を単なる一時的な出力ではなく、永続化された「思考ログ」として管理し、コアメモリと相互作用させます。
- **理由**: 
    - **Salvage（サルベージ）**: 推論の各ステップにおいて、関連する過去の知識の**要約とID**を自動的に検索し、エージェントに提示します。詳細が必要な場合、エージェントは自ら `fetch_details` ツールを使用してその知識を取得します。これにより、コンテキストの肥大化を防ぎつつ、エージェントの意思に基づいた「能動的な想起」を促します。
    - **Accretion（蓄積/蒸留）**: 推論セッションから重要な知見を抽出し、自動的に Graph や Bank へ反映（蒸留）します。増分抽出時には「既存のコンテキスト」を考慮させ、冗長な知識生成を防ぎます。抽出結果はJSONバリデーションによって厳格に検証され、スキーマ整合性を担保します。
    - **Reasoning Provenance**: 「なぜその結論に至ったのか」という思考の軌跡を永続化することで、セッションを跨いだ論理の一貫性維持と、後からの論理監査を可能にします。

### 2-6. 知識の不整合（コンフリクト）の検知と人間による承認 (HITL)
- **方針**: 新しい情報が既存の知識と矛盾する場合、AI が自動的に不整合を検知し、即座に主記憶へ書き込むのではなく「承認待ち（Conflict Center）」へステージングします。
- **理由**: 
    - **SSoT (Single Source of Truth) の品質維持**: 誤った情報や一時的なハルシネーションによって、チームで共有される「黄金の知識ベース」が汚染されるのを防ぎます。
    - **人間による最終判断**: 知識の「進化（方針変更）」なのか「誤り（ノイズ）」なのかの最終的な判別を人間に委ねることで、システムの信頼性を担保します。
    - **透明性の確保**: ダッシュボードを通じて、AI が何を「矛盾」と判断したかを可視化し、知識の更新プロセスをブラックボックス化させないようにします。

### 2-7. コンテキストの密度最適化（Signal-to-Noise Ratio）
- **方針**: MCPツールを介した検索結果の返却数を、常に推論に最適な「Top 10〜20件」程度に厳選し、必要に応じて内容を要約・切り出し（Truncation）します。
- **理由**: 
    - **推論精度の維持**: 大量の情報を一度に提示すると、LLMの「情報の埋没（Lost in the Middle）」を引き起こし、推論精度が低下します。高シグナルな情報のみを届けることで、エージェントの判断能力を最大化します。
    - **システム安定性**: 通信レイヤーでのタイムアウトやメモリ不足を防ぎ、商用グレードの安定性を確保します。
    - **コストと速度**: トークン消費量を抑制し、レスポンスのレイテンシを最小化します。

### 2-8. ドキュメント戦略：日英併記（English Primary + 日本語セクション）
- **方針**: README をはじめとする主要ドキュメントは、英語をメイン構造としつつ、日本語の専用セクションを併設する構成を採用します。
- **理由**: 
    - **発見可能性（Discoverability）**: GitHub の検索エンジンや `awesome-mcp-servers` リスト、海外コミュニティからの流入を確保するため、公用語である英語をベースとします。
    - **共感と転換率（Conversion）**: 日本のAI駆動開発者に対し、冒頭の日本語メッセージで「自分の課題を解決するツールだ」という直感的な共感（AHA体験）を提供します。
    - **成功パターンの踏襲**: `Deno` や `Localsend` など、グローバルで成功しているOSSが採用している「英語メイン＋母国語サポート」のハイブリッドモデルに準拠します。

### 2-9. マルチエージェント認証と変更履歴（Audit）
- **方針**: 複数のAIエージェントやツールが同一の Ripen を共有する際、「どのエージェントが、いつ、何を記録したか」を正確に特定し、ログとして残します。
- **実装**:
  - `data/auth.json` による API キーベースの認証ゲートウェイ。
  - クライアントごとに割り当てられた `SHARED_MEMORY_ACCOUNT` を `audit_logs` および `thought_history` に紐付け。
  - セキュリティと利便性のバランスを考慮し、環境変数によるAPIキーの注入を標準とする。
- **理由**: 
  - **責任の明確化**: AIエージェントが誤った情報を記録した場合、どのセッション・どのエージェントに起因するかを追跡可能にすることで、デバッグと知識の修正を容易にします。
  - **チーム開発の基盤**: 将来的に複数の人間開発者が異なるAIツールを併用するチーム環境において、知識の「出所（Provenance）」を保証します。

### 2-10. 定量的評価指標：LongMemEval
- **方針**: 記憶の「想起精度」と「推論レイテンシ」を定量的に評価するための独自のベンチマークスイート `LongMemEval` を運用します。評価ロジックには、業界標準である **RAGAS (RAG Assessment)** のメトリクスを採用します。
- **理由**: 
    - **客観性と信頼性の担保**: 独自のスコアリングではなく、RAGAS（Faithfulness, Context Recall 等）を用いることで、外部からも納得感のある品質評価を可能にします。
    - **デグレードの早期発見**: プロバイダー（Gemini/Ollama）やモデルを変更した際に、記憶の検索精度が低下していないかを客観的に検証します。
    - **パフォーマンスの最適化**: 12ms 級の超低レイテンシを維持しつつ、Recall@10（上位10件の正解率）を 90% 以上に保つための設計ガイドラインとして機能させます。

### 2-11. 動的なコンテキスト圧縮とトークン管理 (Dynamic Context Compression)
- **方針**: LLMのコンテキストウィンドウ制限（特に500/413エラーの要因となるペイロード過大）を防ぐため、API送信前にトークン数を事前計算し、制限に近づいた場合は自律的に「情報の蒸留（Distillation）」を行う圧縮レイヤーを設けます。
- **実装**:
  - `config.py` にて、推論用（Generation）と圧縮用（Compression）のモデルプールを独立して定義・管理。
  - プロンプト送信前に `count_tokens` を用いてサイズを評価し、安全閾値（例: 制限の90%）を超過した場合、圧縮用モデルによる再帰的な要約処理を自動でトリガー。
- **理由**: 
    - **システムの堅牢性（Resiliency）**: 情報量が増大してもサーバーがクラッシュすることなく、エラーを自己修復して処理を継続できるようにするため。
    - **意味の保存（Semantic Preservation）**: 単純な文字数カット（Truncation）ではなく、LLM自身に「事実の抽出」をさせることで、重要な文脈や論理構造を維持したまま安全なトークンサイズへとダウンサイズします。
    - **コストと速度の分離**: 将来的に「圧縮には高速・安価なモデル（Flash等）」、「推論には高精度モデル（Pro等）」といった戦略的なルーティングを可能にするため。

### 2-12. 思考プロセスの厳格なID管理と整合性保証 (Thought Integrity)
- **方針**: Sequential Thinking における各思考ステップ（`thought_number`）を、セッション内で一意（Unique）な識別子として厳格に管理し、意図しない重複やデータの不整合を物理的に遮断します。
- **実装**:
  - **データベース制約**: `thought_history` テーブルの `(session_id, thought_number)` に対して `UNIQUE` インデックスを適用し、DBレベルでの二重登録を防止。
  - **多重防御（Defense in Depth）**: アプリケーション層で挿入前に既存IDをチェックし、重複時には詳細なエラーメッセージ（修正なら `is_revision=True` を使用せよ、等）をエージェントに返却。
  - **下位互換性の維持**: 既存の重複データがある環境でも起動を妨げないよう、マイグレーション時には警告付きのフォールバック・ロジックを採用。
- **理由**: 
    - **推論の完全性（Provenance）**: 同じ番号の思考が複数存在すると、修正（Revision）や分岐（Branching）の対象が不明確になり、推論の軌跡が破壊されるため。
    - **メタデータの保護**: 重複IDによる上書きが発生すると、過去の思考に紐付いていた検索結果や統計情報といったメタデータが汚染されるリスクを排除するため。
    - **エージェントへのフィードバック**: 「状態管理の誤り」を即座にエラーとして返すことで、AIエージェント自身に正しいプロトコルへの自己修正を促し、より高度な自己組織化を実現するため。

### 2-13. 配信とオンボーディングの設計 (Distribution & Onboarding UX)
- **方針**: 「`uvx ripen --sse` の1コマンドでハブサーバーが起動する」体験を最優先とし、導入の摩擦を極限まで排除します。設定は対話型ウィザード（`ripen-init`）で案内し、IDE登録は `ripen-register` で自動化します。
- **実装**:
  - PyPI パッケージ `ripen` として配布。`uvx`（一時実行）と `pip install`（永続）の両方に対応。
  - `ripen-init hub`: データ保存先、LLMプロバイダー、APIキー、トランスポートモードを対話形式で設定し、`~/.ripen/config.json` に永続化。完了後に「子機用接続URL」を表示。
  - `ripen-init client`: Hub URLを入力するだけで、全IDEへの自動登録を完了。Python不要。
  - `ripen-register --hub-url <url>`: 単独で子機登録のみを実行するコマンドライン版。
  - ネイティブバイナリ（PyInstaller）と Docker イメージによる追加の配布チャネル。
- **理由**: 
  - **Time to First Shared Memory の最小化**: チームメンバーが共有記憶へ初めてアクセスするまでの時間を3分以内に短縮します。
  - **2種類のユーザー体験の分離**: 「サーバーを立てる人（Hub）」と「接続するだけの人（Client）」では必要な設定量が根本的に異なります。この非対称性をCLIで明示的に扱うことで、全ユーザーにとって最適化された体験を提供します。
  - **「名前の資産価値」**: PyPI における短い一般名詞（`ripen` = 5文字）は、ドメイン名と同様に希少性があり、プロジェクトの認知と信頼に直結します。

### 2-14. ゼロコンフィグ起動とGraceful Degradation
- **方針**: LLMプロバイダーやAPIキーが未設定の状態でも、基本的な検索・保存機能は動作する「ゼロコンフィグモード」をデフォルトとします。
- **実装**:
  - 設定の優先順位: 環境変数 > `~/.ripen/config.json` > ハードコードされたデフォルト値。
  - LLM未設定時: 知識の蒸留（Distillation）とAIベースの要約は無効化されるが、FTS5全文検索、ベクトル検索（FastEmbed）、グラフ操作、Memory Bank は完全に動作。
  - サーバー起動時に、現在の構成（LLMプロバイダー、Embeddingエンジン、データ保存先、Dashboard URL、**Client接続URL**）をサマリ表示。
- **理由**: 
  - **「動かない」から「とりあえず動く」へ**: 初回起動時にエラーで挫折するユーザーを撲滅します。
  - **段階的な機能拡張**: ユーザーは基本機能を体験してから、必要に応じてLLMを追加設定できます。
  - **企業環境への対応**: セキュリティポリシーで外部APIの利用が制限される環境でも、ローカル完結で即座に導入可能です。

### 2-15. Hub & Client モデル（マルチエージェント・マルチユーザー知識共有）
- **方針**: Ripenは **「親機（Hub）」** と **「子機（Client）」** の2つの導入形態を明確に区別します。親機は知識の権威（Authority）として機能し、子機はHTTP経由で親機に接続するだけで知識を共有できます。
- **実装**:
  - **親機（Hub）**: `ripen --sse --port 8377` で起動。SSEエンドポイント（`/sse`）とDashboard（`/dashboard`）を公開。データはローカルのSQLite + FAISSで完全管理。
  - **子機（Client）**: Pythonのインストール不要。IDEのMCP設定に `npx mcp-remote <hub-url>/sse` を追加するだけ。`ripen-init client` または `ripen-register --hub-url` で自動設定。
  - **同時接続**: N台のIDEが同一のHubに並行接続し、全員が同一のDB（SQLite WALモード）を読み書きする。
  - **Audit by Agent**: `auth.json` のAPIキーで各クライアントを識別し、誰がいつ何を書いたかを `audit_logs` に記録。Dashboard の「Active Agents」セクションでリアルタイム可視化。
- **理由**: 
  - **根本的な差別化**: 他の全てのMCPメモリサーバーは `stdio`（1:1接続）であり、物理的に知識を共有できません。SSEハブ方式は、この問題を解決する**唯一の実用的なアーキテクチャ**です。
  - **チームのスケールと限界**: 現状のアーキテクチャ（SQLite WALモード）は、個人開発〜小規模チーム（2〜3人）での同時アクセスに最適化されています。高スループットなネットワーク書き込みを伴う大規模な分散チームインフラとしては機能しません。
  - **チーム共有基盤への第一歩**: 個人ツールとしての「便利さ」を超え、少人数チームの「AI駆動開発の共有基盤」としての価値を提供します。将来的には大規模向け分散DB（Turso等）への移行を視野に入れます。
  - **子機のゼロコスト参加**: 子機の導入に Python も `ripen` のインストールも不要なため、技術的なハードルが最も低い状態でチーム全員が恩恵を受けられます。

### 2-18. 商用ライセンスとエコシステムの持続可能性 (Sustainability & Cryptographic Trust)
- **方針**: AGPL-3.0 によるオープンソースの公開性を維持しつつ、プロフェッショナルなチーム・企業利用に対しては、堅牢なデジタル署名（Ed25519）に基づいた商用ライセンス認証を導入します。
- **実装**:
  - **Keygen.sh 統合**: 業界標準のライセンス管理プラットフォームを使用し、マシンのフィンガープリンティングとアクティベーションを管理。
  - **Ed25519 オフライン検証**: サーバーからのレスポンスに HTTP Signature (Ed25519) を付与し、クライアント側で数学的に検証。一度アクティベートすれば、オフライン環境でも「正当な商用版」として動作し続ける Local-First な信頼性を担保。
  - **180日間プロフェッショナル試用**: チームでの導入障壁を下げるため、半年間の長期試用期間をデフォルトで提供。
- **理由**: 
  - **持続可能な開発**: ユーザー数の増大に伴う管理コストやサポート責任に備え、プロジェクトを長期的に維持・進化させるための標準的な運用モデルとして導入。
  - **エンタープライズ要件への対応**: 企業ユーザーが求める「ソースコードの改ざん防止」や「利用統計の管理」を、暗号技術によって透明性高く実現するため。

### 2-16. エッジLLMを導入せず「外部プロバイダー連携」を優先した理由 (Plan A への転換)
- **方針**: Ripen Hub 内部に LLM 推論エンジン（ONNX Runtime 等）を直接埋め込むのではなく、外部の信頼できるプロバイダー（Ollama または Gemini）への接続・検知を優先します。
- **理由**:
    - **JSON Mode の信頼性とモデルサイズ**: Ripen の核となる「知識の蒸留（エンティティ抽出）」には、厳格な JSON フォーマット出力が不可欠です。軽量な 3B〜8B クラスのエッジモデルでも JSON 出力は可能ですが、実用的な精度と「JSON Mode（強制的な構造化出力）」の堅牢性を確保するには、現状では 26B 前後、あるいはそれ以上の規模のモデルが事実上の下限となります。これらをローカルで安定稼働させるには 16GB 以上の専有 RAM と強力な GPU が必要となり、軽量なバックグラウンドサービスとしての Ripen のコンセプト（16GB RAM のノート PC でも動作）と矛盾します。
    - **リソース競合の回避**: IDE（Cursor/VSCode）やビルドプロセスと並行して動作する Hub が、自ら重い LLM 推論を実行すると、CPU/RAM の奪い合いが発生し、最も重要な SSE 配信のレイテンシを悪化させるリスクがあります。
    - **依存関係のクリーン化（DLL Hell の防止）**: 推論ランタイムの埋め込みは、特定の OS/CPU 命令セットへの依存や、ライブラリの競合を引き起こしやすく、インストール時のトラブル（UX の低下）に直結します。
- **FastEmbed を優先採用した理由**:
    - **海馬としての責務**: 記憶サーバーにおいて「検索（ベクトル検索）」は心臓部です。FastEmbed（BGE モデル等）は 100MB 前後と極めて軽量でありながら、CPU 上で瞬時に起動し、SOTA 級の検索精度を提供します。
    - **LLM による代用の非効率性**: LLM を使ってベクトル（Embedding）やタグ１０ほど構築してもらい、タグ検索をすることで代用することは技術的には可能ですが、1つの思考を保存するたびに数 GB の LLM をメモリに展開・推論させるのはリソース効率が極めて悪いです。特化型の軽量モデル（FastEmbed）を内蔵し、高負荷な「思考」のロジックのみを外部（Ollama 等）へ委ねることで、システムの安定性と応答性を最大化しています。

### 2-17. 摩擦ゼロのナレッジ蓄積とHubによる価値増幅 (Zero-Friction Distillation & Hub Amplification)
- **方針**: エージェントが意識的に保存操作を行わなくても、`sequential_thinking` を使って「考える」だけで、バックグラウンドの蒸留器（Distiller）が自動的に事実を抽出し、ローカルHub経由で全エージェントに同期させます。
- **理由**:
    - **UXの魔法（Zero-Friction）**: 従来のナレッジ管理ツールの最大の弱点は「人間やAIが意識して書かなければならない（摩擦がある）」ことでした。Ripenでは、AIがタスクをこなす過程の「つぶやき」から、裏側で自動的に砂金（確定知識）だけが抽出・保存されます。ユーザーもAIも、これまでの行動を一切変えることなく、システムが自律的に進化（Ripen）していきます。
    - **群知能の実現（Swarm Intelligence）**: この「摩擦ゼロで抽出された確定知識」が、ローカルホストのSSE Hubを通じて、別のアカウントや別のIDE（Cursor, Windsurf, Claude Codeなど）で動く他のエージェントにも即座に共有されます。「Agent Aが苦労して解決したバグの解決策を、Agent Bが即座に知っている」という、単一エージェントでは不可能なAIチーム全体のIQ底上げを実現します。これがRipenの提供する真のコアバリューです。

## 3. 非目標（何を、なぜ行わないのか）

### 3-1. 自律的な推論（Reasoning）の実装
- **非目標**: サーバー自体が知識を組み合わせて新しい推論を行うことはしません。
- **理由**: 本サーバーは「海馬（記憶を司る部位）」に特化すべきであり、「前頭前野（思考を司る部位）」は呼び出し側（エージェント）の責任であるためです。責任の分離（SoC）を徹底します。

### 3-2. ペシミスティックな書き込み拒絶（ゲートキーピング）
- **非目標**: 既存の知識との矛盾を理由に、新しい情報の保存を拒絶（ブロック）することはしません。
- **理由**: 
    - **進化の阻害防止**: サーバーは「真偽の絶対的な審判」ではなく、あくまで「観測の記録者」であるべきだからです。情報の正当なアップデートや方針転換をシステムが妨げるリスクを排除します。
    - **パフォーマンスの優先**: 書き込み時の厳密なバリデーションはレイテンシを増大させます。矛盾は「エラー」として排除するのではなく、後続の蒸留プロセスやエージェントの推論において「解決すべき課題」として提示する方針をとります。

### 3-3. グラフ構造の動的な可視化
- **非目標**: リアルタイムでノードとエッジが動く複雑なネットワークグラフ表示機能は持ちません。
- **理由**: インフラとしての機能性に注力するためです。可視化が必要な場合は、ダッシュボード上のリスト形式や、外部ツールへのエクスポート機能を優先します。

### 3-4. ヒューリスティックな重み付け検索
- **非目標**: 「"Error" という文字が含まれていたらスコアを+10する」といった、ハードコードされた経験則（ヒューリスティック）は実装しません。
- **理由**: 現代の埋め込みモデルは十分に文脈を理解しています。手動の重み付けは将来のモデル改善の恩恵を阻害し、メンテナンスコスト（技術負債）を増大させるためです。

### 3-5. 複雑なマルチユーザー権限管理
- **非目標**: 粒度の細かいユーザー別アクセス制御（RBAC）などは実装しません。
- **理由**: 本サーバーは個人、あるいは特定のプロジェクトチームの「専属記憶」としての利用を想定しています。過度な複雑さはパフォーマンスとポータビリティを損なうためです。

### 3-6. 無制限なデータ返却（Unbounded Retrieval）
- **非目標**: 「過去の全履歴を一度にリストアップする」といった機能を、エージェントが利用するMCPツールとしては提供しません。
- **理由**: 
    - **役割の分離**: MCPツールは「推論とアクション」のための実行レイヤーであり、膨大なデータの監査や俯瞰的な分析は、別途構築される「管理用UI（管理レイヤー）」の責務であるためです。
    - **認知負荷の制御**: 全履歴の提示はエージェントにとってのノイズとなり、目的の達成を阻害するリスクがあるためです。

### 3-7. 完全非同期化による極限のレイテンシ最適化 (Over-Optimization for Latency)
- **非目標**: セッション終了時の最終的な知識蒸留（Final Distillation）において、エージェントへのレスポンス速度を数秒縮めるためだけの「完全な非同期化（バックグラウンドへの完全分離）」は行いません。
- **理由**: 
    - **Read-After-Writeの一貫性**: エージェントが思考を終えた直後のタスクで過去の記録を検索した際、「さっき結論づけたはずの知識」がまだDBに反映されていない状態（ハルシネーションの誘発）を防ぐため、セッションの境界では確実に状態を同期させます。
    - **ライフサイクルの安全性**: MCPサーバーが一時的なプロセスとして実行される環境において、バックグラウンドタスクが完了する前にプロセスが終了（SIGTERM等）し、知識が永久にロストするリスクを排除します。数秒のレスポンス向上よりも、データの確実な永続化（信頼性）を優先します。

### 3-8. 特定のクラウドベンダーへのロックイン
- **非目標**: Google Gemini 専用のシステムとして固定することはしません。
- **理由**: ユーザーのセキュリティポリシーやコスト、速度の要件に合わせて、ローカル LLM (Ollama) や将来的な他社 API へ柔軟に切り替えられる「プロバイダー非依存（Agnostic）」なアーキテクチャを維持するためです。

## 4. 設計哲学
**「複雑さよりも信頼性、機能数よりも整合性、個人ツールよりもチームの持続可能性」**

エージェントが迷ったとき、常に「そこに行けば正しい過去の記録がある」と確信できる場所であることが、Ripen の根本的な存在意義です。

そして、その「確信」は、**チーム全員のエージェントが同じ場所を参照し、かつそのシステムが将来にわたってメンテナンスされ続けるという「信頼」** があるときにこそ、最大の価値を発揮します。
