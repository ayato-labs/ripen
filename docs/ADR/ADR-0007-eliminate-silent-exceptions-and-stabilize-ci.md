# ADR-0007: 例外の握りつぶし排除とCI環境におけるサーバー起動の安定化

- **Date**: 2026-06-13
- **Status**: Accepted
- **Deciders**: Gemini CLI (Agent), ayato-labs (User)

## Context
システム全体の可観測性（Observability）と信頼性を向上させるため、以下の2つの課題に対処する必要がありました。

1.  **例外の握りつぶし**: コードベース内の数箇所で `except: pass` というパターンが使用されており、予期せぬエラーが発生してもログに残らず、原因究明が困難になっていました。
2.  **CI環境でのシステムテストの不安定さ**: GitHub ActionsなどのCI環境において、サーバー起動テスト（`test_server_http_connectivity`）が、「正常終了（コード0）」を返して即座に終了してしまう現象が発生していました。これは、仮想環境における標準入力（stdin）の切断がサーバー終了のトリガーとして誤検知されていたためです。

## Decision
1.  **「例外握りつぶし」の完全排除**:
    - 全ソースコードを精査し、`except: pass` を `logger.debug` や `logger.exception` による適切な状況記録に置き換えました。
    - 特に、サーバーのシャットダウン処理、ポートのクリーンアップ処理、およびCLIの終了プロンプトにおける例外処理を改善しました。
2.  **システムテスト用フィクスチャの強化**:
    - サーバープロセス起動時に `stdin=subprocess.DEVNULL` を指定し、標準入力の切断による意図しない終了を防止しました。
    - `PYTHONUNBUFFERED=1` および `python -u` を使用し、ログのバッファリングを無効化してCI上でリアルタイムに状況を確認可能にしました。
    - `stderr=subprocess.STDOUT` を指定し、標準エラー出力を標準出力に統合してデバッグを容易にしました。
3.  **Lintルールの遵守**:
    - 未使用のインポート（`signal`）を削除し、Ruffの `UP041`（`asyncio.TimeoutError` から組み込み `TimeoutError` への移行）を適用しました。

## Consequences
### Positive
- **デバッグ効率の向上**: エラーが発生した際、ログから正確な原因（ファイル名、行番号、エラー種別）を特定できるようになりました。
- **CIの安定化**: システムテストが環境に左右されず安定してパスするようになり、開発のVelocityが向上しました。
- **コード品質の向上**: モダンなPython（3.11+）のベストプラクティスに準拠し、技術負債を削減しました。

### Negative / Risks
- **ログの増大**: `logger.debug` を追加したことにより、開発時のログ出力がわずかに増加しますが、適切にローテーション（retention=2）されているため運用上の問題はありません。
