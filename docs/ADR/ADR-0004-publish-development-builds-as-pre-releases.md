# ADR-0004: 開発版バイナリの開発版プレリリース（dev-latest）としての自動配信

- **Date**: 2026-05-24
- **Status**: Accepted
- **Deciders**: ayato-labs (Human), Antigravity (Agent)

## Context

`ADR-0003` により、`develop` ブランチおよびプルリクエスト（PR）時にもバイナリビルドを実行し、GitHub Actions のビルドアーティファクト（Artifacts）として保存することが決定されました。

しかし、この運用には以下の実用的な課題が残っていました：

1. **ダウンロードの摩擦**: Actions の Artifacts からファイルをダウンロードするには、GitHub へのログインが必須であり、かつ対象のワークフロー実行履歴を都度検索する手間が発生するため、テストの容易性が損なわれていました。
2. **製品版との混同**: ビルドされるバイナリ名が正式リリース版と同じ（`Ripen.exe` 等）であるため、開発者が現在ローカルで動かしているバイナリが「どのバージョンの開発版なのか」「本番製品版なのか」を判別しにくく、バグ検証時の混乱を招くリスクがありました。

本番リリースのクリーンさ（本番アセット一覧の整理）を維持しつつ、開発者が「常に最新の開発版バイナリを簡単に1クリックで取得できる場所」を確保する必要があります。

## Decision

開発版ビルドの識別名を変更し、GitHub Releases の「プレリリース（Pre-release）」枠を利用して自動上書き配信する運用を導入します。

1. **ファイル名の変更**:
   - `develop` ブランチおよび PR 時のビルド成果物の名前を `Ripen-dev.exe` および `RipenInit-dev.exe` に変更します。
2. **`dev-latest` プレリリースの新設と自動更新**:
   - `develop` ブランチへのプッシュ時に、GitHub Actions を介して `dev-latest` というプレリリース（Pre-release: true）を自動的に作成・更新します。
   - すでに `dev-latest` リリースが存在する場合は、既存の `Ripen-dev.exe` / `RipenInit-dev.exe` アセットを自動で上書き（Overwrite）アップロードします。これにより、常に最新の `develop` バイナリが同一の固定リンクからダウンロード可能になります。
3. **プルリクエスト（PR）時の動作**:
   - 複数人による同時開発時の上書き競合を防ぐため、PR 段階のビルド成果物はプレリリースへはデプロイせず、従来通り GitHub Actions の Artifacts への保存のみとします。

## Consequences

### Positive
- **テスト効率の大幅向上**: 開発者は GitHub Releases ページを開くだけで、常に最新の開発版バイナリをログイン不要・1クリックでダウンロードできます。
- **製品版と開発版の明確な識別**: ファイル名に `-dev` が付与されるため、実行中のバージョン混同を防止できます。
- **本番リリースの整合性維持**: プレリリース（橙色の Pre-release バッジ）として配信されるため、一般ユーザーが誤って本番用と誤認してダウンロードするリスクが最小限に抑えられます。

### Negative / Risks
- `develop` へのプッシュのたびにリリースアセットの上書き API リクエストが発生しますが、GitHub API の制限範囲内であり実用上の支障はありません。
