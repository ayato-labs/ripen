# ADR 0001: Docker配布の取りやめとWindows .exeへの一本化

## ステータス

承認済み (Accepted)

## 日付

2026-05-24

## コンテキスト (背景)

Ripen はこれまで開発チーム間での知識共有を目的とし、マルチプラットフォーム向けに Docker イメージのビルドおよび配布を行っていました。しかし、以下の要因から配布方針の見直しが必要となりました。

1. **企業・商用環境におけるライセンス問題**
   Docker Desktop などの有償化に伴い、企業や商用環境で Docker を導入・利用する際にライセンス上のコンプライアンス課題（利用規模に応じた有償契約の必要性）が発生し、開発者にとって導入の障壁となっていました。
2. **実質的なマルチOS対応の完了**
   Ripen は常駐型の HTTP/SSE (Streamable HTTP) サーバーとして動作します。チーム内に Windows 環境が1台でも稼働していれば、Mac や Linux など他のOSを利用するメンバーの AI エージェント（Cursor, Claude Code, Antigravity など）は、HTTP 通信を介してその親機に接続するだけで利用できます。
   そのため、各クライアント端末のOSごとに Docker コンテナを個別に立ち上げる必要性が極めて低く、Windows 環境があれば十分にマルチプラットフォーム運用が可能です。

## 意思決定

Docker によるコンテナ配布方針を取りやめ、配布形態を以下の2点に一本化・統合します。

1. **Windows Nativeバイナリ (`.exe`) の配布**
   PyInstaller によってビルドされた `ripen-hub.exe` および `ripen-admin.exe` を GitHub Releases を通じて提供します。
2. **Python ソースコードからの起動**
   macOS や Linux を利用する開発者、あるいはソースコードから動作させたいユーザー向けに、`uv` パッケージマネージャー等を利用した Python ソースからの直接実行をサポートします。

これに伴い、GitHub Actions の CI/CD パイプラインから Docker のビルド・プッシュ (`build_docker`)、ZIPアセットの作成 (`build_zip_assets`)、および Docker 環境下での Linux テスト (`test_linux`) ジョブを削除しました。

## 影響

- **メリット**:
  - 企業内でのライセンスコンプライアンス上の懸念がなくなり、導入が容易になります。
  - CI/CD パイプラインから重い Docker ビルド処理が削除され、CI 実行時間が大幅に短縮（開発ベロシティの向上）します。
  - 配布成果物の構成がシンプルになり、メンテナンスコストが低下します。
- **デメリット**:
  - Linux 環境でサーバーを常時ホストさせたい場合に、Docker Compose での即時立ち上げができなくなり、Pythonソースからのセットアップが必要になります。
