# Ripen アーキテクチャ詳細

本ドキュメントは、Ripenの内部構造、データフロー、および主要な設計パターン（State Governance）について詳細に解説します。

## 1. システム全体構成 (High-Level Architecture)

Ripenは、AIエージェントの長期記憶（Hippocampus）として機能するローカルホスト型のMCP（Model Context Protocol）サーバーです。

## 2. コア設計パターン (Core Design Patterns)

### 2-1. Compute-then-Write パターン
マルチエージェント環境におけるSQLiteのロック競合を防ぐための最重要パターンです。
- **課題**: 埋め込みベクトルの計算（LLM API呼び出しやローカル推論）は遅く、これをDBのトランザクション内で実行すると、他のエージェントからの読み書きがブロック（`database is locked`）されてしまいます。
- **解決策**: トランザクションの外で事前に必要なAI計算（Embedding等）を全て完了させ、データベースへの書き込み処理自体は軽量化してミリ秒単位（< 50ms）でアトミックに実行します。

### 2-2. ハイブリッド検索 (FTS5 + Vector Similarity)
検索精度と速度を両立させるため、以下の2段構えの検索を実装しています。
1. **FTS5 (Full-Text Search)**: 独自のトークナイザー（またはN-gram）を用いたキーワードの完全一致検索。IDや特定の関数名の検索に威力を発揮します。
2. **Semantic Search (Vector)**: 意味的な類似度による検索。`FastEmbed` や `Google Gemini` 等を使用してベクトル化されたデータを比較します。

### 2-3. Sequential Thinking の厳格なID管理 (Reasoning Provenance)
エージェントの思考プロセスを記録し、論理の一貫性を保つための基盤です。
- **重複検知と遮断**: `thought_history` テーブルに `(session_id, thought_number)` の `UNIQUE` 制約を適用。さらにアプリケーション層での事前チェックにより、IDの重複送信を物理的に遮断します。
- **Revision と Branching**: 単なる上書きを許さず、明示的な `is_revision=True` フラグや `branch_id` を要求することで、「なぜ思考が変更されたか」という由来（Provenance）を保証します。

## 3. 知識のライフサイクルと成熟度 (Knowledge Maturity)

システムに保存された知識は、時間とともに減衰（Decay）し、アクセスされることで強化されるアルゴリズムを持っています。

### 3-1. パラメータの定義
- **Stability (安定性)**: 情報の内容が変化しない度合い。頻繁にコンフリクト（事実の不一致）が起こる情報は安定性が下がります。
- **Importance Score (重要度)**: 検索でヒットし（Salvage）、実際にエージェントによって利用された（Accretion）回数に応じて増加します。
- **Decay Rate (減衰率)**: 最後にアクセスされてからの時間経過に伴い、重要度を徐々に低下させる係数。

### 3-2. 蒸留 (Distillation) と GC (Garbage Collection)
- **Incremental Distillation**: 思考ステップのたびに、バックグラウンドで一時的なコンテキストから重要な事実（Entity/Relation）を抽出します。
- **Final Distillation**: セッション終了時に履歴全体を俯瞰し、高シグナルな知識を構造化して主記憶（Graph/Bank）に定着させます。
- **Garbage Collection**: 減衰が進み、閾値を下回った一時的なノイズは、バックグラウンドメンテナンス（`run_maintenance_logic`）により自動的に「非アクティブ（Archive）」状態に移行します。

## 4. プロトコルの安定化パッチ (Protocol Resilience)

MCPプロトコル（JSON-RPC）の制約や、特定のクライアント（Claude Desktop等）の挙動に起因する切断を防ぐため、深層的なパッチを当てています。

- **Permissive Handshake**: `InitializeRequest` が遅延・欠落した場合でも、一定のタイムアウト後に強制的に `Initialized` 状態へ移行させ、セッションのハングを防ぐモンキーパッチ (`ServerSession._received_request`)。
- **Type Sanitization**: MCP SDKが厳格に要求する型（Integerなど）に対し、エージェントが誤って文字列（String）を送ってきた場合でも、再帰的に型を変換してバリデーションエラーを吸収する `_sanitize_mcp_dict` 関数。
- **Extreme Guard (STDOUT Redirect)**: `stdio` トランスポート使用時、ライブラリ等が意図せず `print`（標準出力）を行った瞬間にJSON-RPCプロトコルが破壊されるのを防ぐため、`sys.stdout = sys.stderr` による強圧的なリダイレクトを実施しています。

## 5. データベーススキーマの概要

物理ファイルは役割ごとに分割されています。

1. **`knowledge.db`**: コアとなる記憶（Graph, Bank, Observations）。マルチアクセスが集中するため、WALモードで稼働。
2. **`thoughts.db`**: Sequential Thinking専用の履歴データ。一時的だが大容量のデータが書き込まれるため分離。
3. **`master.db`**: システム全体のステータス管理、マイグレーション履歴の記録。

*詳細なテーブル定義については、`reference/schema.md` を参照してください。*
