# MCP Tool Descriptions Visibility Fix (2026-05-09)

## 症状 (Symptoms)
- AIエージェントがMCPツールの説明（Description）を読み取れず、ツールの用途を正しく理解できない。
- クライアントUI上でツールの説明欄が空、または「This is a tool from...」といったデフォルトの定型文のみが表示される。
- サーバー側でdocstringを記述しているにもかかわらず、メタデータとして反映されない。

## 原因 (Causes)
1. **docstring抽出の不安定性**:
   - `fastmcp` または `mcp` SDKのバージョンにより、docstringのパース挙動が変化。
   - 特に3連クォート (`"""`) の直後が改行で始まっている場合、抽出結果が空になるケースがある。
2. **ゾンビプロセスの影響 (Windows特有)**:
   - サーバーを再起動したつもりでも、古いプロセスがポート（8377等）を占有し続け、古いコード（修正前）をサーブし続けていた。
   - Windowsの `taskkill` が権限やタイミングにより失敗し、サイレントに古いプロセスが生き残る。

## 解決策 (Resolution)
1. **明示的なメタデータ定義**:
   - `@mcp.tool()` デコレータに `description` 引数を追加し、docstringに依存せず直接説明文を指定。
   - `FastMCP()` コンストラクタに `instructions` 引数を追加し、サーバー全体の役割を定義。
2. **プロセスの強制終了**:
   - ポートを占有しているプロセスを `taskkill /F /PID <PID>` で確実に終了させる。
3. **SDKパッチの調整**:
   - 内部的なJSON-RPCのバリデーションやリダイレクト設定が、メタデータの伝播を阻害していないか確認（今回は主にプロセスの入れ替え不足が主因）。

## 予防策 (Prevention)
- 今後のツール追加時も、docstringとは別にデコレータへの `description` 指定を必須とする。
- サーバー起動ログに `instructions` やツール数を出力し、最新のコードが読み込まれているか視覚的に確認できるようにする。
- Windows環境での開発時は、`netstat` 等でポートの解放を明示的に確認する習慣をつける。
